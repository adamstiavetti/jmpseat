# First Release MVP

Working name note: "Deadhead Club" is a working product name pending legal and trademark clearance. This document does not claim legal or trademark clearance.

Product principle: Utility first. Community second. Social feed last.

Identity principle: Verified privately. Anonymous publicly. Accountable internally.

## 1. Purpose

This document defines the practical first release MVP for Deadhead Club based on the current blueprint, MVP scope, research, interviews, operating model, build sequencing, trust and safety rules, and beta-readiness constraints.

It exists to resolve a recurring ambiguity in the planning set:

- the broader written V1 feature envelope
- versus the smaller first release that should actually ship first

This document treats the first release MVP as the authoritative answer to "what should the first real release be?"

## 2. First Release Definition

The first release MVP is a private, invite-only, verified airline-worker beta focused on one base across multiple roles.

Its purpose is to prove that verified airline workers will return for trusted Base Board and Layover Board utility inside a safe, gated environment.

It is not a public launch, not a generic social app, not a schedule tool, not a marketplace, and not an AI-led product.

## 3. Core MVP Thesis

The first release should validate this loop:

1. A trusted airline worker creates an account.
2. The user completes or enters aviation-worker verification.
3. The user joins relevant gated boards and rooms tied to airline, role, and base context.
4. The user reads useful base or layover knowledge.
5. The user posts, comments, saves, or searches for practical information.
6. The user returns because the product was useful, trustworthy, and easier than fragmented alternatives.

If this loop does not work, broader V1 expansion is premature.

## 4. MVP Goal

The first release MVP should prove all of the following:

- verified airline workers will use the product for practical utility, not only curiosity
- base-specific and layover-specific knowledge are strong enough to drive repeat use
- gated discussion can feel useful without feeling risky or sketchy
- verification can be completed with acceptable friction
- moderation and safety controls are strong enough for controlled private beta use
- one base across multiple roles can achieve enough density to feel useful

## 5. What The First Release Includes

### Identity and access

- account creation
- login
- basic email verification
- aviation-worker verification through practical V1 methods
- airline / role / base profile
- public handle
- internal accountability behind public identity
- invite-only or access-gated beta participation

### Verification

The preferred first-release verification paths are:

- basic email verification
- aviation work email verification where practical
- manual non-upload review paths first
- controlled manual upload review only if private storage and deletion controls are ready

Peer vouching may be tested later as a supplemental signal, but should not be the primary unknown-user verification path for the first release.

### Community utility surfaces

- Base Boards
- city and airport Layover Boards
- a minimal set of verified rooms tied to airline, role, base, or topic

These are the core product wedge.

### Content capabilities

- posts
- comments
- saves
- search

These capabilities exist to support the utility loop, not to create a feed-first product.

### Safety and operations

- reporting flow
- moderation review workflow
- admin moderation actions
- aviation-worker verification review workflow
- admin verification decisions
- emergency escalation category for safety or privacy issues

## 6. What The First Release Does Not Need

The first release should not require:

- public launch
- nationwide community density
- all room types from long-range planning docs
- advanced automation
- monetization infrastructure
- polished social-feed mechanics
- native mobile
- roster or calendar integrations
- airline portal integrations
- verification vendor APIs

## 7. Stretch Features That Are Not Core First Release

These may still belong inside the broader V1 planning envelope, but they should not define first-release success:

- Jumpseat Brief AI layover planner
- crew-friendly deals directory
- broader casual/social rooms
- career tools
- advanced board summaries
- monetization layers

If included early, they should be treated as secondary support features, not the reason the MVP exists.

## 8. Primary MVP Surfaces

### Base Boards

Base Boards are the strongest strategic wedge for the first release.

They should help users answer recurring practical questions such as:

- commuting
- reserve life
- line tips
- parking
- transit
- food and errands
- new-hire survival
- base culture
- contract and legality discussion where allowed

### Layover Boards

Layover Boards are strongly validated and should focus on practical crew intelligence such as:

- food
- cheap bites
- happy hour options
- walking-distance suggestions
- transportation
- wellness and downtime
- airport-area tips
- safety notes without exact hotel exposure

### Verified rooms

Verified rooms should be present, but kept narrow and useful.

They are important because:

- users compare them to trusted Facebook groups
- they support candid discussion
- they reinforce the verified-worker trust boundary

They should not become a generic social feed in the first release.

## 9. Trust and Safety Requirements

The first release MVP is only viable if these constraints are enforced:

- no exact crew hotel addresses or exact hotel exposure
- no trip or schedule screenshots
- no passenger private information
- no airport security procedures
- no live operations-sensitive information
- no live crew tracking
- no doxxing
- no harassment
- no impersonation
- no unsafe meetup pressure
- no vendor spam

Anonymous participation must remain accountable internally.

Real anonymous or member-generated participation must not be opened without verification, reporting, moderation, admin controls, and emergency escalation capability.

## 10. UX Standard For The First Release

The actual in-app MVP should feel:

- familiar
- clear
- easy to use
- low-friction
- trustworthy
- not gimmicky
- not obviously AI-generated

The public cinematic waitlist experience and the private app experience should be treated differently.

The public waitlist can be visual and branded. The actual MVP app should optimize for utility and trust, not cinematic flourish.

## 11. First Release Community Shape

The recommended first release community is:

- one base
- across multiple roles
- with controlled invitations
- with seeded useful content
- with active admin coverage

This is preferable to:

- launching everywhere
- one role across many bases
- broad public signups without density

The reason is simple: density and usefulness matter more than geographic breadth in the first release.

## 12. Success Metrics For The First Release

The first release should be judged by:

- verification completion
- useful posting and commenting
- search and save behavior
- repeat visits
- useful Base Board contributions
- useful Layover Board contributions
- user trust
- manageable moderation load

Success should not be judged mainly by:

- vanity signup volume
- generic social activity
- requests for excluded risky features
- broad public awareness

## 13. Broad V1 Versus First Release

The repo contains a broader V1 scope that includes:

- Jumpseat Brief
- deals
- wider utility surface expansion
- more complete room taxonomy

That broader V1 is still useful as a planning envelope.

But the first release MVP should be understood as narrower:

- verified identity and access
- Base Boards
- Layover Boards
- minimal verified rooms
- posts / comments / saves / search
- moderation and verification operations

When there is tension between the broad V1 list and the smallest practical safe release, the first release MVP should win.

## 14. Final Definition

The first release MVP for Deadhead Club is:

A tightly controlled, invite-only, verified airline-worker beta for one base-centered multi-role community, built to validate whether Base Boards, Layover Boards, and gated utility discussion can create repeat-use, trusted aviation-worker knowledge exchange inside a safe, moderated environment.

## 15. Operating Rule

Future MVP decisions should be checked against this question:

Does this make the first-release utility loop safer, clearer, more useful, or easier to validate?

If not, it probably belongs after the first release.
